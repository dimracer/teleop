# teleop

Прямое (без ROS/ROS2) управление связкой **AgileX Bunker Mini 2.0** (мобильное шасси) +
**AgileX PiPER** (6-DOF роборука) с одного штатного пульта Bunker Mini — FlySky FS-i6X.

Идея: оператор рулит платформой как обычно (левый джойстик + SWB), а два переключателя
пульта, которые прошивка Bunker Mini физически не использует (SWA и SWD), плюс правый
джойстик и крутилка VRA — отданы под управление рукой. Никакой доработки железа не
требуется: чейзи и так транслирует состояние всех каналов пульта в CAN-шину (кадр `0x241`),
мы просто её слушаем.

Все скрипты общаются с железом напрямую через `python-can` (socketcan) и официальные
SDK производителя — `piper_sdk` и `pyagxrobots`. ROS/ROS2 не используется.

## Файлы

### `agilex_platform.py`
Базовые классы-обёртки над официальными SDK:

- `BunkerMiniController` — обёртка над `pyagxrobots.pysdkugv.BunkerBase` (включение CAN,
  задание скорости/поворота, чтение состояния базы).
- `PiperArmController` — обёртка над `piper_sdk.C_PiperInterface_V2`: включение/выключение
  руки, движение по углам суставов (`move_joints`, умеет ждать реального прихода в позу через
  `wait_settle=True`), движение в декартовых координатах (`move_to_pose`), ручная донастройка
  позы (`begin_jog`/`apply_jog`), хват из текущей (уже подогнанной) позы без автоспуска
  (`grasp_and_lift`) и старый вариант со спуском (`descend_grasp_lift`, `pickup_sequence`),
  управление захватом (`gripper`), аварийный останов (`emergency_stop`).
- `AgileXPlatform` — объединяет базу и руку в один объект с safety-обвязкой
  (`startup`/`shutdown`/`emergency_stop_all`).

Все методы работают в режиме `dry_run=True` по умолчанию — команды только логируются,
в CAN ничего не уходит. Это позволяет проверить логику без подключённого железа.

`enable()`/`enable_joint_mode()`/`enable_end_pose_mode()` не просто шлют команду один раз —
они ПОДТВЕРЖДАЮТ результат по фидбеку руки (`GetArmEnableStatus()`/`GetArmStatus()`) и
повторяют попытку при неудаче, вместо того чтобы молча считать, что всё включилось (см.
раздел «Диагностика» ниже).

Импортируется остальными скриптами, самостоятельно не запускается (кроме встроенной
демонстрации в `if __name__ == "__main__"`).

### `bunker_rc_reader.py`
Читает «сырые» CAN-кадры Bunker Mini напрямую (в обход `pyagxrobots`, у которой чтение
пульта не реализовано):

- кадр `0x241` — состояние пульта: тумблеры SWA/SWB/SWC/SWD, оба джойстика, крутилка VRA;
- кадр `0x221` — фактическая линейная/угловая скорость шасси (нужно для safety-проверки
  «база стоит на месте» перед тем, как дать руке начать захват).

Даёт класс `BunkerRcReader` с подпиской на события `on_swd_down(...)` / `on_swa_down(...)`
(срабатывают один раз на переключение тумблера, не на удержание) и метод `is_stationary()`.
Тоже импортируется, самостоятельно не запускается (кроме демо-заглушки в
`if __name__ == "__main__"`).

### `calibrate_approach.py`
Отдельная утилита для подбора позы `APPROACH` (см. раздел «Калибровка» ниже). Подключается
к руке ТОЛЬКО на чтение (не включает CAN-режим, не шлёт команд движения), в реальном
времени печатает углы суставов, пока рука в drag-teach и её двигают руками. Ctrl+C —
зафиксировать текущую позу и напечатать готовую строку `APPROACH = JointAngles(...)`.

### `pult_pickup_teleop.py`
Главный исполняемый скрипт — сама логика «подъехал → взял». Собирает вместе
`BunkerRcReader` и `PiperArmController` в конечный автомат:

```
IDLE --SWD--> JOGGING --SWA--> HOLDING --SWA--> TURNING --(авто)--> IDLE
          ^______________SWD (сброс донастройки)____|
```

- **IDLE → JOGGING** (SWD вниз): рука выходит в заранее откалиброванную позу `APPROACH`
  над предметом (и ЖДЁТ подтверждения по фидбеку, что реально пришла — `wait_settle=True`,
  см. «Диагностика #2» ниже), включается режим ручной донастройки.
- **в JOGGING**: правый джойстик (влево-вправо / вверх-вниз) двигает захват по X/Y,
  крутилка VRA — по высоте Z. Это нужно, если база подъехала не идеально в то же место
  и под тем же углом, что при калибровке — донастройка компенсирует разницу «на глаз».
  VRA (Z) — это и есть спуск к предмету: подводи джойстиком/VRA вплотную, отдельного
  автоматического спуска перед хватом больше нет. Смещение ограничено безопасной коробкой
  ±40 мм по X/Y и ±50 мм по Z вокруг `APPROACH`.
- **JOGGING → HOLDING** (SWA вниз): захват закрывается ПРЯМО ИЗ ТЕКУЩЕЙ (уже подогнанной
  джойстиком/VRA) позы и ДЕРЖИТ — без автоспуска и без автоподъёма
  (`PiperArmController.grasp_hold`; автоподъём убран, см. «Диагностика #4»).
- **HOLDING → TURNING → IDLE** (SWA вниз повторно): разворот-сброс — J1 → 150°,
  спуск через J2/J3 на 40%, отпустить захват, уход в zero point
  (`PiperArmController.turn_lower_release_home`, см. раздел «Разворот-сброс» ниже).
- **SWD вниз повторно, находясь в JOGGING**: рука ФИЗИЧЕСКИ возвращается в базовую позу
  `APPROACH` (`move_joints(APPROACH, wait_settle=True)`) и донастройка начинается заново
  с нуля (раньше это только «забывало» текущий сдвиг координат, никуда руку не двигая —
  см. «Диагностика #2»).

Перед выполнением команд на реальном железе скрипт проверяет `rc.is_stationary()` — если
база ещё физически движется, действие игнорируется с предупреждением в лог (программно
остановить базу через CAN нельзя, пока включён пульт — у него наивысший приоритет).

## Требования

Не устанавливались автоматически — ставится вручную:

```bash
pip3 install python-can piper_sdk pyagxrobots
```

Python 3.8+.

## Железо и подключение

- **Bunker Mini 2.0**: свой USB-to-CAN адаптер (идёт в комплекте), CAN-шина на битрейте
  **500000**.
- **PiPER**: свой отдельный USB-to-CAN адаптер (тоже в комплекте), CAN-шина на битрейте
  **1000000**. Смешивать оба устройства в одну физическую шину нельзя — разный битрейт.
- Оба адаптера подключаются к **одному компьютеру** (два свободных USB-порта). Так как
  CAN — проводная шина, этот компьютер должен физически ехать на платформе (мини-ПК на
  борту), а не быть отдельным ноутбуком в стороне.
- Пульт FS-i6X по CAN не подключается — он работает по своему радиоканалу 2.4 ГГц
  напрямую с приёмником внутри Bunker Mini. Скрипт лишь «подслушивает» состояние пульта
  через CAN-телеметрию шасси.

### Поднятие CAN-интерфейсов

Из комплекта `piper_sdk` (`find_all_can_port.sh`, `can_activate.sh`, `can_muti_activate.sh`):

```bash
# узнать, какой USB-адаптер к какому порту подключён
bash find_all_can_port.sh

# поднять оба интерфейса с нужными именами и битрейтами
bash can_activate.sh can_bunker 500000  "<usb-port-bunker>"
bash can_activate.sh can_piper  1000000 "<usb-port-piper>"
# или сразу оба через can_muti_activate.sh (см. документацию piper_sdk)
```

## Калибровка позы `APPROACH`

В `pult_pickup_teleop.py` есть заглушка:

```python
APPROACH = JointAngles(j1=0, j2=40, j3=-60, j4=0, j5=20, j6=0)
```

Это joint-поза «рука висит над предметом издалека» — отправная точка, от которой потом
идёт ручная донастройка джойстиком.

Порядок калибровки (делай именно в этом порядке — нарушение порядка, в частности пункта 5,
было причиной нестабильной работы, см. «Диагностика» ниже):

1. Подъезжаешь на роботе примерно туда, откуда обычно будешь подъезжать к предмету.
2. На руке — ОДИН клик по кнопке между J5/J6 → загорится ровный зелёный (drag-teach,
   запись). Руками подводишь захват в позу approach.
3. Запускаешь `python3 calibrate_approach.py --piper-can can_piper` — он только читает,
   команд движения не шлёт. Смотришь на экран, углы обновляются в реальном времени.
4. Как устроит — `Ctrl+C`. Скрипт печатает готовую строку `APPROACH = JointAngles(...)` —
   вставляешь её в `pult_pickup_teleop.py` вместо заглушки.
5. **ОБЯЗАТЕЛЬНО, ДО того как отключать питание руки**: ещё раз клик по той же кнопке
   между J5/J6 и убедись, что индикатор ПОГАС (не горит, не мигает). Это останавливает и
   ЗАВЕРШАЕТ запись drag-teach. Если обрезать питание руки прямо во время записи (индикатор
   ещё горит), есть риск, что во внутренней памяти руки останется недозаписанная/случайная
   траектория, которая потом неожиданно проиграется в CAN-режиме.
6. Только теперь верни руку в её «нулевую точку» (zero point, см. PiPER Quick Start Manual,
   раздел 2.2) и можно выключать/включать питание.
7. После включения питания — НЕ нажимай кнопку между J5/J6 повторно, если индикатор уже не
   горит. Лишний клик может по ошибке запустить повторную запись, а двойной клик — включить
   режим ВОСПРОИЗВЕДЕНИЯ записанной ранее траектории (индикатор замигает) — в этом режиме
   рука может начать двигаться сама по себе независимо от CAN-команд.
8. Только теперь запускай `pult_pickup_teleop.py --live`.

## Запуск

Проверка логики без железа (эмулирует нажатия пульта и джойстик, ничего не отправляет в CAN):

```bash
python3 pult_pickup_teleop.py --demo
```

Реальный запуск на борту робота:

```bash
python3 pult_pickup_teleop.py --live --bunker-can can_bunker --piper-can can_piper
```

Флаги:

- `--bunker-can` — имя поднятого CAN-интерфейса базы (по умолчанию `can_bunker`)
- `--piper-can` — имя поднятого CAN-интерфейса руки (по умолчанию `can_piper`)
- `--poll-timeout` — таймаут чтения CAN-кадра в секундах (по умолчанию 0.05)

Остановка — `Ctrl+C`: рука уводится в home и выключается корректно.

## Управление с пульта (шпаргалка)

| Канал пульта | Штатная функция | Что делает в этом проекте |
|---|---|---|
| S1 (левый стик, вперёд-назад) | газ | не тронут, рулит базой как обычно |
| S2 (левый стик, влево-вправо) | поворот | не тронут |
| SWB | пульт/CAN-режим базы | не тронут |
| SWC | свет | не тронут |
| SWD | не используется базой | approach + вход в донастройку / сброс донастройки |
| SWA | не используется базой | 1-е нажатие: захват (без автоподъёма); 2-е: разворот-сброс (J1→150°, спуск J2/J3 на 40%, отпустить, zero point) |
| правый стик | не используется базой | донастройка X/Y захвата |
| VRA (крутилка) | не используется базой | донастройка высоты Z захвата |

## Диагностика: рука реагирует не при каждом запуске / нажатии

Симптом: скрипт в логе честно печатает переходы состояний («>>> IDLE -> JOGGING...»,
«descend_grasp_lift: старт/готово»), но физически рука не двигается — и это происходит не
всегда, а через раз, без явной системы.

Как диагностировали (кратко): сняли `candump can_piper` во время момента, когда рука не
пошевелилась после SWD/SWA. В дампе оказались ТОЛЬКО фреймы фидбека от руки (`0x2A1-0x2A8`,
`0x251-0x256`, `0x261-0x266`) и НИ ОДНОГО управляющего фрейма (`0x151` ModeCtrl, `0x155-0x157`
JointCtrl, `0x152-0x154` EndPoseCtrl, `0x471` EnableArm). Это значит, что в тот момент хост
вообще ничего не передал на шину — дело не в том, что рука проигнорировала команду, а в том,
что команда физически не ушла с компьютера.

До этого фикса код слал `EnableArm()`/`ModeCtrl()` ОДИН раз и сразу считал, что всё
получилось, никак не проверяя фидбек. Если конкретно этот единственный CAN-фрейм терялся
(а сразу после power-cycle руки или при работе одновременно двух USB-CAN адаптеров на одном
хосте это случается) — рука оставалась в standby/не в том move_mode, но Python продолжал как
ни в чём не бывало слать `JointCtrl`/`EndPoseCtrl`, которые рука просто не принимала, а сам
код никак не показывал, что что-то пошло не так.

**Что исправлено в коде** (`agilex_platform.py`):

- `enable()` теперь дополнительно шлёт `MasterSlaveConfig(0xFC, 0, 0, 0)`, явно фиксируя
  роль «исполнительная рука» — исключает неоднозначность master/slave-конфигурации.
- `enable()`/`enable_joint_mode()`/`enable_end_pose_mode()` ждут подтверждения по
  `GetArmEnableStatus()` / `GetArmStatus().ctrl_mode`/`mode_feed` и повторяют попытку до
  3 раз; если так и не подтвердилось — кидают `RuntimeError` с понятным сообщением вместо
  того, чтобы молча продолжать.
- `JointCtrl`/`EndPoseCtrl`/`GripperCtrl` теперь оборачивают реальный вызов SDK в
  try/except и явно логируют, если отправка не удалась, вместо того чтобы падать без следа.

**Что стоит проверить/изменить в процедуре работы (см. также «Калибровка» выше)**:

- Не обрезай питание руки, пока индикатор drag-teach ещё горит — сначала останови запись
  явным кликом, убедись что индикатор погас, и только потом выключай питание.
- После включения питания не кликай по кнопке между J5/J6, если индикатор уже не горит —
  лишний клик/двойной клик может включить повторную запись или воспроизведение сохранённой
  траектории (индикатор мигает), которая будет конфликтовать с CAN-командами.
- Резкий провал руки примерно на 30 мм по одной из осей сразу после SWA — это ПОХОЖЕ именно
  на проигрывание случайно сохранённой (при обрыве питания во время записи) внутренней
  траектории руки, а не на баг в `descend_grasp_lift()` (там опускание идёт на явно заданные
  `descend_mm`/`lift_mm`, отсчитанные от текущей позы, никаких скачков в 30 мм по логике кода
  быть не должно).
- Если проблема повторится: держи открытым второй терминал с `candump can_piper` во время
  работы `pult_pickup_teleop.py --live`. Если в момент SWD/SWA в дампе появляются `0x151`/
  `0x155-0x157`/`0x152-0x154` — команда ушла, и дело уже не на стороне Python/CAN-отправки,
  а в состоянии/режиме самой руки. Если фреймов по-прежнему нет — смотри лог: теперь при
  сбое отправки или неподтверждённом enable/mode там будет явное сообщение об ошибке.

## Диагностика #2: включение стало надёжным, но остались 4 других симптома

После фикса из раздела выше рука стала стабильно включаться и выполнять команды при
каждом запуске (это подтверждено). Но отдельно от проблемы «не включилось» остались
4 симптома, которые оказались обычными логическими багами в самом сценарии
(`pult_pickup_teleop.py`), а не проблемой передачи CAN-команд:

**Честно про повторный `candump`**: во втором приложенном дампе (снятом во время УСПЕШНОЙ
работы — рука реально двигалась и хватала) управляющих фреймов (`0x151`/`0x152-154`/
`0x155-157`/`0x471`) в дампе тоже не оказалось видно — только те же ID фидбека, что и в
первом дампе. Раз рука в этот момент точно двигалась, это значит, что предположение о
конкретных ID управляющих кадров, сделанное в прошлый раз по манулу, нельзя считать
подтверждённым этим дампом (в Quick Start Manual таблицы с точными ID кадров вообще нет —
она бралась из общих знаний о протоколе). Плюс во втором дампе нет меток времени, так что
сопоставить конкретный кадр с конкретным нажатием SWD/SWA всё равно было бы нельзя.
Поэтому диагноз ниже — по прочтению исходного кода (`pult_pickup_teleop.py`/
`agilex_platform.py`), а не по CAN-дампу.

1. **SWD переводит руку в разные конечные позы вместо стабильной `APPROACH`.**
   Причина: код делал `move_joints(APPROACH)`, затем **фиксированную** `time.sleep(1.0)`,
   и только потом `begin_jog()` считывал текущую позу как «базу» для донастройки. Если
   движение руки не успевало закончиться за эту секунду (а время движения зависит от того,
   откуда рука стартовала — из home или из предыдущей позы — и не гарантировано меньше
   секунды) — `begin_jog()` считывал позу «на лету», в случайной точке traektorii, и база
   для донастройки каждый раз получалась чуть разной.
   **Фикс**: `move_joints(..., wait_settle=True)` теперь реально ждёт (опрашивая
   `GetArmJointMsgs()`) прихода в целевые углы (допуск 1.5°, таймаут 5 с) вместо слепой
   паузы, и только после этого `begin_jog()` фиксирует базу.

2. **Правый джойстик/VRA иногда не двигают руку.**
   Прямых доказательств по CAN не было (см. оговорку выше), поэтому это — наиболее
   вероятная причина по структуре кода: `apply_jog()`/`move_to_pose()` слали
   `EndPoseCtrl()` без подтверждения и без повторной проверки, что рука до сих пор
   реально в режиме MOVE L — код просто один раз проверял режим при входе в джог и потом
   бесконечно доверял своей локальной переменной, даже если реальный режим руки где-то
   незаметно «уехал» (кратковременный сбой связи, watchdog в прошивке и т.п.).
   **Фикс**: `apply_jog()` теперь каждые `JOG_MODE_RECHECK_S` (0.5 с) сверяет реальный
   режим по `GetArmStatus()` и, если он разошёлся с ожидаемым MOVE L, — молча
   переотправляет `ModeCtrl`. Плюс `EndPoseCtrl`/`JointCtrl`/`GripperCtrl` теперь при
   ошибке отправки делают один быстрый повтор вместо немедленного отказа.

3. **SWA сначала опускает руку вниз и только потом хватает — а не должен, если джойстик
   уже подвёл захват к предмету.**
   Причина: `descend_grasp_lift()` всегда опускался на фиксированные `descend_mm` от
   текущей позы перед хватом. Но раз высота (Z) уже управляется вручную через VRA во
   время JOGGING, это давало ДВОЙНОЙ спуск — сначала вручную джойстиком, потом ещё
   автоматически на 60 мм вниз поверх этого.
   **Фикс**: `_on_swa_down` теперь вызывает новый метод `grasp_and_lift()` — закрывает
   захват прямо из текущей (уже подогнанной) позы и поднимается, без дополнительного
   автоспуска. Старый `descend_grasp_lift()` остался в коде (на случай, если понадобится
   сценарий, где донастройка останавливается ВЫШЕ предмета и нужен отдельный автоспуск).

4. **Повторный SWD в JOGGING (сброс донастройки) никак не двигает руку.**
   Причина: код вызывал только `begin_jog()`, который лишь заново запоминает ТЕКУЩУЮ
   (уже сдвинутую джойстиком) позу как новый «ноль» — реального движения к `APPROACH`
   никто не командовал, поэтому рука физически не шевелилась, хотя в логе это выглядело
   как «сработавшее» событие.
   **Фикс**: повторный SWD теперь вызывает тот же код, что и первый вход в JOGGING —
   `move_joints(APPROACH, wait_settle=True)` + `begin_jog()` — то есть реально возвращает
   руку в `APPROACH`, дожидаясь подтверждения, и только потом даёт донастройку заново.

Если после этих фиксов какой-то из симптомов повторится на реальном стенде — самое
полезное, что можно прислать: лог терминала (`teleop_log`) ЗА ТОТ ЖЕ ЗАПУСК, где явно видно
предупреждения `[PiPER] джог: реальный режим руки разошёлся...` или
`move_joints: рука не подтвердила приход...` (они появятся именно в момент сбоя, если
причина в рассинхронизации режима/незавершённом движении).

## Диагностика #3: SWD не всегда доводит руку в APPROACH (чинится 2–3-м нажатием)

Симптом после фиксов #2: при нажатии SWD рука иногда вообще не едет в `APPROACH`.
Повторное нажатие SWD перезапускает перемещение, и со 2–3-го раза рука доезжает.

**Причина.** `wait_settle=True` в `move_joints()` только ЖДАЛ прихода в позу по фидбеку,
но сама команда `JointCtrl` отправлялась ровно один раз. Успешная запись кадра в
CAN-сокет не гарантирует, что он дошёл до прошивки: потерянный/проигнорированный кадр
не вызывает исключения — рука просто стоит. Код честно ждал 5 с, писал warning
(`рука не подтвердила приход...`) и шёл дальше: захватывал ФАКТИЧЕСКУЮ (неправильную)
позу как базу джога и входил в JOGGING. Повторный SWD «чинил» это лишь потому, что
переотправлял ту же команду. Дополнительно `enable_joint_mode()` доверял
закэшированному `self._move_mode` и не сверял реальный режим руки по фидбеку — тот же
класс бага, что уже чинили в `apply_jog()` (Диагностика #2, симптом 2), но в
joint-ветке он оставался.

**Фикс** (`agilex_platform.py`, `move_joints(wait_settle=True)` теперь сам добивается
результата и возвращает `bool`):

1. Перед каждой попыткой реальный режим руки сверяется по `GetArmStatus()` (а не по
   кэшу); при расхождении `ModeCtrl(MOVE J)` переотправляется с подтверждением.
2. После отправки `JointCtrl` идёт контроль фидбека: если суставы вообще не стронулись
   за `MOVE_NO_PROGRESS_TIMEOUT_S` (1.2 с) — кадр считается потерянным и команда
   переотправляется сразу, не дожидаясь полного таймаута.
3. Всего до `MOVE_SEND_RETRIES` (3) попыток, каждая с полным `MOVE_SETTLE_TIMEOUT_S`
   (5 с) на доезд. Успех — только по подтверждению `GetArmJointMsgs()` (допуск 1.5°).

Если и после всех ретраев поза не подтверждена (обрыв CAN, механическое препятствие) —
`move_joints` возвращает `False` и пишет **error** в лог; `_enter_jogging()` в
`pult_pickup_teleop.py` (по согласованию) всё равно входит в JOGGING от фактической
позы — как раньше, но теперь это громко видно в логе, и SWD можно нажать ещё раз.

Логика ретраев покрыта тестом `test_move_retry.py` (фейковый piper, эмуляция потери
кадров): доставка с 1/2/3 попытки, отказ после трёх потерь, пересинхронизация режима,
мгновенный выход если рука уже в позе.

## Диагностика #4: ретраи не помогли — прошивке нужна пара ModeCtrl+команда

Тест на стенде показал: фикс из «Диагностики #3» (переотправка одиночного `JointCtrl`)
симптом НЕ убрал — SWD по-прежнему доводил руку в `APPROACH` только со 2–3-го нажатия.
Плюс «вернулся» рывок: после SWA рука резко уходила вверх.

**Уточнённая причина.** Кадры не теряются — прошивка их **игнорирует**, если перед
командой движения не было свежего `ModeCtrl`. Улики:

- Повторное нажатие SWD помогало не из-за переотправки самого `JointCtrl`, а потому,
  что между нажатиями `begin_jog()` переводил руку в MOVE L — и на следующем SWD код
  был вынужден заново послать `ModeCtrl(MOVE J)`. Сразу после свежего `ModeCtrl`
  команда выполнялась. На первом нажатии режим уже «MOVE J» — `ModeCtrl` не слался,
  и одиночный `JointCtrl` молча игнорировался.
- Официальные примеры piper_sdk шлют `MotionCtrl_2`/`ModeCtrl` перед **каждой**
  командой движения, в каждой итерации цикла — а не один раз при «смене режима».
- «Рывок вверх» после SWA — то же отложенное исполнение: команда подъёма на 80 мм
  (`grasp_and_lift`) исполнялась не в момент отправки, а когда прошивка наконец
  «активировалась», и выглядела как резкий скачок.

**Фикс** (вторая итерация, `agilex_platform.py`):

1. Пара «`ModeCtrl` + команда движения» теперь отправляется **всегда вместе** — и в
   `move_joints` (MOVE J), и в `move_to_pose` (MOVE L), как в официальных примерах SDK.
2. При `wait_settle=True` пара переотправляется каждые `MOVE_RESEND_PERIOD_S` (1 с),
   пока фидбек `GetArmJointMsgs()` не подтвердит приход в позу (переотправка позиционной
   команды безопасна); до `MOVE_SEND_RETRIES` (3) внешних попыток по 5 с.
3. Автоподъём после захвата **убран совсем** (по согласованию): SWA из JOGGING теперь
   вызывает `grasp_hold()` — только закрыть захват и держать. Старый `grasp_and_lift()`
   оставлен в коде на будущее.

## Разворот-сброс: второй SWA (состояние TURNING)

Новая логика второго нажатия SWA (из HOLDING). Вместо «отпустить и домой» выполняется
автоматическая последовательность `turn_lower_release_home()`:

1. **Разворот**: J1 → `TURN_J1_TARGET_DEG` = **150°** (согласовано: ровно 180°
   физически недоступно, предел PiPER ±154°, программный лимит ±150°; знак константы
   поменяй, если сбрасывать надо в другую сторону). Остальные суставы держатся как
   при захвате.
2. **Спуск**: J2/J3 увеличиваются по модулю на `DROP_LOWER_PCT` = **40%**
   (согласованный диапазон 30–50%) — рука опускается к месту сброса.
3. **Отпускание**: захват открывается на 60 мм.
4. **Zero point**: рука уходит в нулевую позу (`move_home`) → состояние IDLE.

Каждый переезд идёт через `move_joints(wait_settle=True)` — с парой ModeCtrl+JointCtrl,
периодической переотправкой и подтверждением по фидбеку. Цикл состояний теперь:

```
IDLE --SWD--> JOGGING --SWA--> HOLDING --SWA--> TURNING --(авто)--> IDLE
          ^______________SWD (сброс)_____|
```

Всё покрыто обновлённым `test_move_retry.py` (7 проверок): пара ModeCtrl перед каждым
JointCtrl, «капризная» прошивка (игнор без свежего ModeCtrl), потеря кадров, отказ,
рука уже в позе, порядок и целевые углы разворота-сброса (150°/56°/−84°, отпускание
между спуском и уходом домой).

## Безопасность

- Максимальный груз PiPER — 1.5 кг, максимальный груз Bunker Mini 2.0 — 25 кг.
- Ход захвата PiPER — 0-70 мм.
- Перед enable() руки на реальном стенде — она должна быть в нулевой точке,
  drag-teach остановлен (и завершён ДО отключения питания, см. «Калибровка»).
- Пока пульт включён, он имеет наивысший приоритет над CAN-командами движения базы —
  скрипт не может «силой» остановить платформу, только читает её фактическую скорость
  и не даёт руке начать движение, пока база не остановится сама.
- Ручная донастройка позы ограничена безопасной коробкой вокруг `APPROACH`
  (`PiperArmController.JOG_XY_LIMIT_MM` / `JOG_Z_LIMIT_MM`).

## Известные ограничения / что можно добавить дальше

- Нет распознавания объекта — донастройка позы полностью ручная (джойстиком, на глаз).
  Полностью автоматическая компенсация угла подъезда потребует камеры и детекции объекта
  (например, метка ArUco/AprilTag на предмете + пересчёт координат камера → база руки).
- Полноценное непрерывное телеуправление всеми 6 суставами руки с пульта Bunker Mini
  физически невозможно (не хватает свободных каналов) — для этого нужен отдельный
  контроллер (геймпад/джойстик на хосте или штатный master-arm PiPER).
