# teleop

Прямое (без ROS/ROS2) управление связкой **AgileX Bunker Mini 2.0** (мобильное шасси) +
**AgileX PiPER** (6-DOF роборука) с одного штатного пульта Bunker Mini — FlySky FS-i6X.

Идея: оператор рулит платформой как обычно (левый джойстик + SWB), а два переключателя
пульта, которые прошивка Bunker Mini физически не использует (SWA и SWD), плюс правый
джойстик и крутилка VRA — отданы под управление рукой. Никакой доработки железа не
требуется: чейзи и так транслирует состояние всех каналов пульта в CAN-шину (кадр `0x241`),
мы просто её слушаем.

Все три скрипта общаются с железом напрямую через `python-can` (socketcan) и официальные
SDK производителя — `piper_sdk` и `pyagxrobots`. ROS/ROS2 не используется.

## Файлы

### `agilex_platform.py`
Базовые классы-обёртки над официальными SDK:

- `BunkerMiniController` — обёртка над `pyagxrobots.pysdkugv.BunkerBase` (включение CAN,
  задание скорости/поворота, чтение состояния базы).
- `PiperArmController` — обёртка над `piper_sdk.C_PiperInterface_V2`: включение/выключение
  руки, движение по углам суставов (`move_joints`), движение в декартовых координатах
  (`move_to_pose`), ручная донастройка позы (`begin_jog`/`apply_jog`), готовый сценарий
  «опуститься-схватить-поднять» (`descend_grasp_lift`, `pickup_sequence`), управление
  захватом (`gripper`), аварийный останов (`emergency_stop`).
- `AgileXPlatform` — объединяет базу и руку в один объект с safety-обвязкой
  (`startup`/`shutdown`/`emergency_stop_all`).

Все методы работают в режиме `dry_run=True` по умолчанию — команды только логируются,
в CAN ничего не уходит. Это позволяет проверить логику без подключённого железа.
Импортируется остальными двумя скриптами, самостоятельно не запускается (кроме
встроенной демонстрации в `if __name__ == "__main__"`).

### `bunker_rc_reader.py`
Читает «сырые» CAN-кадры Bunker Mini напрямую (в обход `pyagxrobots`, у которой чтение
пульта не реализовано):

- кадр `0x241` — состояние пульта: тумблеры SWA/SWB/SWC/SWD, оба джойстика, крутилка VRA;
- кадр `0x221` — фактическая линейная/угловая скорость шасси (нужно для safety-проверки
  «база стоит на месте» перед тем, как дать руке начать захват).

Даёт класс `BunkerRcReader` с подпиской на события `on_swd_down(...)` / `on_swa_down(...)`
(срабатывают один раз на переключение тумблера, не на удержание) и метод `is_stationary()`.
Тоже импортируется, самостоятельно не запускается (кроме демо-заглушки в
`if __name__ == "__main__"`).

### `pult_pickup_teleop.py`
Главный исполняемый скрипт — сама логика «подъехал → взял». Собирает вместе
`BunkerRcReader` и `PiperArmController` в конечный автомат:

```
IDLE --SWD--> JOGGING --SWA--> HOLDING --SWA--> IDLE
          ^______________SWD (сброс донастройки)____|
```

- **IDLE → JOGGING** (SWD вниз): рука выходит в заранее откалиброванную позу `APPROACH`
  над предметом, включается режим ручной донастройки.
- **в JOGGING**: правый джойстик (влево-вправо / вверх-вниз) двигает захват по X/Y,
  крутилка VRA — по высоте Z. Это нужно, если база подъехала не идеально в то же место
  и под тем же углом, что при калибровке — донастройка компенсирует разницу «на глаз».
  Смещение ограничено безопасной коробкой ±40 мм по X/Y и ±50 мм по Z вокруг `APPROACH`.
- **JOGGING → HOLDING** (SWA вниз): рука опускается из текущей (уже подогнанной) позы,
  закрывает захват, поднимается.
- **HOLDING → IDLE** (SWA вниз): отпускает предмет, возвращается в home.
- **SWD вниз повторно, находясь в JOGGING**: сбросить донастройку и вернуться к базовой
  позе `APPROACH` (если увёл слишком далеко).

Перед выполнением команд на реальном железе скрипт проверяет `rc.is_stationary()` — если
база ещё физически движется, действие игнорируется с предупреждением в лог (программно
остановить базу через CAN нельзя, пока включён пульт — у него наивысший приоритет).

## Требования

Не устанавливались автоматически — ставится вручную:

```bash
pip3 install python-can piper_sdk pyagxrobots
```

Python 3.8+.

## Железо и подключение

- **Bunker Mini 2.0**: свой USB-to-CAN адаптер (идёт в комплекте), CAN-шина на битрейте
  **500000**.
- **PiPER**: свой отдельный USB-to-CAN адаптер (тоже в комплекте), CAN-шина на битрейте
  **1000000**. Смешивать оба устройства в одну физическую шину нельзя — разный битрейт.
- Оба адаптера подключаются к **одному компьютеру** (два свободных USB-порта). Так как
  CAN — проводная шина, этот компьютер должен физически ехать на платформе (мини-ПК на
  борту), а не быть отдельным ноутбуком в стороне.
- Пульт FS-i6X по CAN не подключается — он работает по своему радиоканалу 2.4 ГГц
  напрямую с приёмником внутри Bunker Mini. Скрипт лишь «подслушивает» состояние пульта
  через CAN-телеметрию шасси.

### Поднятие CAN-интерфейсов

Из комплекта `piper_sdk` (`find_all_can_port.sh`, `can_activate.sh`, `can_muti_activate.sh`):

```bash
# узнать, какой USB-адаптер к какому порту подключён
bash find_all_can_port.sh

# поднять оба интерфейса с нужными именами и битрейтами
bash can_activate.sh can_bunker 500000  "<usb-port-bunker>"
bash can_activate.sh can_piper  1000000 "<usb-port-piper>"
# или сразу оба через can_muti_activate.sh (см. документацию piper_sdk)
```

## Калибровка позы `APPROACH`

В `pult_pickup_teleop.py` есть заглушка:

```python
APPROACH = JointAngles(j1=0, j2=40, j3=-60, j4=0, j5=20, j6=0)
```

Это joint-поза «рука висит над предметом издалека» — отправная точка, от которой потом
идёт ручная донастройка джойстиком. Откалибровать под свой стол/предмет:

1. Перевести руку в drag-teach (кнопка между J5/J6, короткое нажатие — начало записи).
2. Руками подвести захват в нужную позицию над предметом.
3. Прочитать текущие углы суставов: `piper.GetArmJointMsgs()`.
4. Подставить эти углы в `APPROACH`.

## Запуск

Перед первым включением руки в CAN-режим убедись, что она физически в нулевой точке
и drag-teach остановлен (индикатор между J5/J6 не горит) — иначе SDK не даст
переключиться в CAN-режим.

Проверка логики без железа (эмулирует нажатия пульта и джойстик, ничего не отправляет в CAN):

```bash
python3 pult_pickup_teleop.py --demo
```

Реальный запуск на борту робота:

```bash
python3 pult_pickup_teleop.py --live --bunker-can can_bunker --piper-can can_piper
```

Флаги:

- `--bunker-can` — имя поднятого CAN-интерфейса базы (по умолчанию `can_bunker`)
- `--piper-can` — имя поднятого CAN-интерфейса руки (по умолчанию `can_piper`)
- `--poll-timeout` — таймаут чтения CAN-кадра в секундах (по умолчанию 0.05)

Остановка — `Ctrl+C`: рука уводится в home и выключается корректно.

## Управление с пульта (шпаргалка)

| Канал пульта | Штатная функция | Что делает в этом проекте |
|---|---|---|
| S1 (левый стик, вперёд-назад) | газ | не тронут, рулит базой как обычно |
| S2 (левый стик, влево-вправо) | поворот | не тронут |
| SWB | пульт/CAN-режим базы | не тронут |
| SWC | свет | не тронут |
| SWD | не используется базой | approach + вход в донастройку / сброс донастройки |
| SWA | не используется базой | захват предмета / отпустить и вернуться домой |
| правый стик | не используется базой | донастройка X/Y захвата |
| VRA (крутилка) | не используется базой | донастройка высоты Z захвата |

## Безопасность

- Максимальный груз PiPER — 1.5 кг, максимальный груз Bunker Mini 2.0 — 25 кг.
- Ход захвата PiPER — 0-70 мм.
- Перед enable() руки на реальном стенде — она должна быть в нулевой точке,
  drag-teach остановлен.
- Пока пульт включён, он имеет наивысший приоритет над CAN-командами движения базы —
  скрипт не может «силой» остановить платформу, только читает её фактическую скорость
  и не даёт руке начать движение, пока база не остановится сама.
- Ручная донастройка позы ограничена безопасной коробкой вокруг `APPROACH`
  (`PiperArmController.JOG_XY_LIMIT_MM` / `JOG_Z_LIMIT_MM`).

## Известные ограничения / что можно добавить дальше

- Нет распознавания объекта — донастройка позы полностью ручная (джойстиком, на глаз).
  Полностью автоматическая компенсация угла подъезда потребует камеры и детекции объекта
  (например, метка ArUco/AprilTag на предмете + пересчёт координат камера → база руки).
- Полноценное непрерывное телеуправление всеми 6 суставами руки с пульта Bunker Mini
  физически невозможно (не хватает свободных каналов) — для этого нужен отдельный
  контроллер (геймпад/джойстик на хосте или штатный master-arm PiPER).
